# Active Ticket

ID: MER-81

Title: Resolve stranded CI/build-hygiene tree changes (ci.yml + bpf build-tags; verify regenerated .o vs pinned clang)

Objective:
A concurrent loop left **uncommitted** build-hygiene changes in the working tree
that must be resolved (committed or partly reverted) so the now-pushed branch's CI
stays green and the tree is clean for further Phase-3 work. The changes:
- `.github/workflows/ci.yml` — add the LLVM apt repo so `clang-${CLANG_VERSION}`
  installs on jammy (CI was failing to install pinned clang);
- `bpf/*.c` (counter, sk_msg, sock_ops, tc_egress, tc_ingress) — a `//go:build
  ignore` line so `go build ./...` stops emitting "C source files not allowed";
- **`bpf/*_bpfel.o`** — regenerated bytecode (this is the risk, see below).

Stay in scope: only these stranded files. Do NOT change BPF program logic, the
frozen ADR-0004 schema, or any unrelated code. Do NOT start MER-71/74/76/80.

Dependencies:
- None functionally. **D10 (pinned clang + `verify-gen`):** the committed `.o` must
  be byte-reproducible by `make ebpf` with the **pinned clang** version; CI's
  `verify-gen` job diffs after regeneration. The stranded `.o` change must be
  re-derived with pinned clang, not adopted blindly.
- Runtime: Lima `meridian` VM (pinned clang 17 + bpftool). Verify in an isolated window.

Acceptance Criteria:
1. **Verify the `.o` regeneration with pinned clang:** on Lima, run `make ebpf` (or
   the bpf2go regen) with the **pinned** clang. Then:
   - if the regenerated `bpf/*_bpfel.o` are **byte-identical** to the stranded
     working-tree `.o` → keep them (the regen is legitimate);
   - if they **differ** (the stranded `.o` came from a wrong/newer clang or a
     non-deterministic build) → **revert the `.o`** to the committed HEAD versions
     and keep only the `.c` `//go:build ignore` tags + `ci.yml`. The `.c` build-tag
     addition must NOT change bpf2go output (clang ignores Go build tags) — confirm.
2. The `//go:build ignore` on `bpf/*.c` makes `go build ./...` / `go vet ./...`
   emit **no** "C source files not allowed" warning, while `make ebpf` still
   regenerates byte-identical bindings (the tag is invisible to clang/bpf2go).
3. `ci.yml`: the LLVM-repo step is correct (keyring + sources.list for the pinned
   `CLANG_VERSION`); `make ebpf` + `make test-bpf` still pass on the CI toolchain.
4. After commit, the working tree is **clean** (no stranded bpf/ci changes);
   `make check-gate-skips` still 10/10 0 skips on Lima; `go build`/`go vet ./...`
   clean and warning-free; `make check-commits` passes (MER-81 ref).
